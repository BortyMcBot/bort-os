### [SEVERITY: MEDIUM] preflight.js output suppression blocklist assessment (DOCUMENT ONLY)
- File: os/preflight.js (lines 272-321)
- Finding: The `enforceHighSensitivityOutput()` function has blocklist patterns for: (1) Bearer tokens/Authorization headers, (2) OpenAI `sk-` keys, (3) Google API keys `AIza...`, (4) GitHub PATs `ghp_`, (5) Google OAuth access tokens `ya29.`, (6) Token key-value pairs (access_token/refresh_token/id_token), (7) Cookie headers, (8) Private key PEM blocks, (9) Large base64 runs (60+ chars), (10) Raw email Received headers. **Gap analysis against current codebase secrets:** The codebase uses the following secret types: `BRIDGE_TOKEN` (Pinchtab bridge token -- format unknown, but likely caught by the base64_run pattern if long enough), `X_CLIENT_ID` / `X_CLIENT_SECRET` / `X_ACCESS_TOKEN` / `X_REFRESH_TOKEN` (caught by `token_kv` pattern when in key=value format, and by `auth_bearer` when in Authorization header), Gmail OAuth tokens (caught by `google_oauth_access` for `ya29.*` access tokens and `token_kv` for refresh_token). **Potential gaps:** (a) AWS-style keys (`AKIA...`) are not in the blocklist but are checked by the PR review secret detector in `pr-review-job.mjs`. (b) Anthropic API keys (pattern `sk-ant-...`) would be caught by the `openai_key` pattern since it matches `sk-`. (c) X Client Secret has no specific pattern but would be caught if it appears as a token_kv pair. (d) If `BRIDGE_TOKEN` is short (<60 chars) and appears as a standalone value (not in Bearer or token_kv format), it might not be caught. **Recommendation:** Add a pattern for generic `BRIDGE_TOKEN` or Pinchtab token format if known.
- Fix applied: no -- os/preflight.js is a dangerous file and must not be modified per review rules. Documented only.
- Dangerous file: yes -- blocked, documented only
