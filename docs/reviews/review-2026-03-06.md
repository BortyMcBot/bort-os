### Quick Wins (low effort, high value)

**[Engagement-driven post type rotation]**
Why: The current `dayRotationType()` uses a fixed day-of-week mod-3 rotation, ignoring which post types actually perform best.
How: Use the engagement log data already captured by `x_engagement_job.js` to weight rotation toward whichever post type (original_take, quote_tweet, link_post) gets the best impressions/likes ratio over the trailing 14 days.
Effort: Low
Relevant skills: (no new skills -- changes to os/x_daily_post.js only)

**[Gmail classify upgrade: LLM-based triage]**
Why: The current `classify.js` uses keyword matching (`looksSpammy`, `looksImportant`) which misses nuanced signals like order confirmations, shipping updates, or eBay-related emails that will matter for EBAY_RESALE_V1.
How: Add a lightweight LLM classify step (via the `gemini` skill or a free-tier model from the allowed list) that takes subject+from and returns a category. Fall back to keyword match if the model call fails or is slow.
Effort: Low
Relevant skills: gemini (already installed, allowed on ops-core and web)

**[X digest: add Anthropic, LangChain, and eBay to curated accounts]**
Why: Bryan's interests include AI/ML and resale ops, but the curated account list has no AI lab beyond OpenAI and no resale/marketplace accounts. Anthropic, LangChainAI, and eBay would directly feed the digest with relevant signal.
How: Add entries to the `curated` array in `os/x_digest_job.js` and optionally to BORT_INTERESTS.md "Curated accounts."
Effort: Low
Relevant skills: (no new skills -- edit to x_digest_job.js)

**[Apple Reminders integration for daily review action items]**
Why: The Gmail daily review classifies and labels but never creates follow-up reminders. Important emails that need a reply or action could automatically generate a reminder.
How: After labeling a thread as Important, use the `apple-reminders` skill (already allowed on inbox hat) to create a reminder with the subject line and a 24-hour due date.
Effort: Low
Relevant skills: apple-reminders (installed, allowed on inbox)

**[Telegram chat ID centralization]**
Why: Already logged as Antipattern 14 in PROJECTS_ACTIVE.md. Three files hardcode chat ID 8374853956.
How: Move the ID to `openclaw.json` or a shared constants module and import it everywhere.
Effort: Low
Relevant skills: (no new skills -- refactor only)

### Pipeline Improvements (medium effort)

**[X post pipeline: draft-preview-approve loop via Telegram]**
Why: Currently x_daily_post.js fires tweets autonomously with no preview. Bryan has no chance to edit or veto before it posts, which is risky for an account building a reputation.
How: Add a two-phase flow: (1) generate draft and send it to Bryan via Telegram (bluebubbles/message tool), (2) wait for an approval command (/approve, /edit, /skip) before posting. Store pending drafts in memory/x_pending_post.json with a TTL.
Effort: Medium
Relevant skills: bluebubbles (installed, allowed on ops-core and autonomous)
Note: This would touch os/x_daily_post.js. No dangerous file changes required.

**[X post content quality: use LLM to generate original takes]**
Why: The current `buildPostFromAnchor` and `buildFallbackTweet` functions produce templated, repetitive content. The fallback tweets are static strings. This limits engagement growth.
How: Feed the top-scored digest entry + topic domains + voice notes from BORT_INTERESTS.md into an LLM call (via `gemini` skill or a Codex model) to generate a tweet draft with Bort's voice. Keep the enforceLength() guard. If the LLM call fails, fall back to the current template approach.
Effort: Medium
Relevant skills: gemini (installed, allowed on web and ops-core)

**[Gmail backlog sweep: auto-unsubscribe pipeline]**
Why: The repo already has `integrations/gmail/unsubscribe.js` and `other-prune.js` but no cron job running them. Subscription bloat is a solved problem waiting to be activated.
How: Wire `unsubscribe.js` into a weekly cron (inbox hat) that processes threads labeled Bort/SpamReview. Use the `extractUnsubTargets` function already in `lib.js` to hit List-Unsubscribe links. Add the sender to `archiveAfterUnsubscribe.senders` in prefs so future emails from them are auto-archived.
Effort: Medium
Relevant skills: (no new skills -- uses existing gmail integration code)

**[Resale pipeline: photo intake via Telegram]**
Why: EBAY_RESALE_V1 notes that "photo intake path (Telegram vs manual upload) still an open question." Telegram is the fastest path since Bryan already interacts with Bort there.
How: Add a Telegram command handler (/sell or /list) that accepts photo attachments, saves them to ~/resale/inbox/<timestamp>/, and triggers item identification via a vision model. Use `nano-banana-pro` (Gemini 3 Pro Image, already on resale hat) for identification and `pinchtab` for eBay comp research once the API is approved.
Effort: Medium
Relevant skills: nano-banana-pro (installed, allowed on resale), pinchtab (installed, allowed on resale), bluebubbles (for Telegram interaction)
Note: Blocked by EBAY_RESALE_V1 -- eBay Developer Program approval pending.

**[Budget dashboard: daily Telegram summary]**
Why: The X budget ledger tracks spend but Bryan has no visibility without SSHing into the VPS. A daily summary via Telegram would surface overspend or underspend patterns.
How: Add a small cron job that reads `memory/x_budget_ledger.json`, computes today's and trailing-7-day spend, and sends a one-line summary via Telegram. Extend later to include model usage costs via the `model-usage` skill.
Effort: Medium
Relevant skills: model-usage (installed, allowed on ops-core)

### OpenClaw Platform Features Not Yet Used

**[MCP (Model Context Protocol) server integration via mcporter]**
Why: The `mcporter` skill is installed and allowed on ops-core, but there is no evidence of any MCP servers being configured or used. MCP servers can provide Bort with structured access to external tools and data sources beyond what individual skills offer -- e.g., database queries, file system access on remote machines, or structured API wrappers.
How: Run `mcporter list` on the VPS to see what's available, then configure relevant MCP servers for Bort's use cases (e.g., a Google Sheets MCP server for resale tracking, or a filesystem MCP server for managing ~/resale/inbox/).
Effort: Medium
Relevant skills: mcporter (installed, allowed on ops-core)

**[Autonomous mode: task queue generation and execution]**
Why: The autonomous runner infrastructure exists (scripts/autonomous_runner.mjs, autonomous_queue.md, autonomous_guard.mjs) but the queue contains only placeholder candidates. The autonomous hat allows coding-agent, github, gh-issues, and openai-image-gen -- enough to do real overnight work like repo maintenance, test coverage improvements, or documentation updates.
How: Build a queue generation step that scans for low-hanging improvements (lint warnings, TODO comments, test gaps) and populates autonomous_queue.md. The runner already has the execution loop; it just needs real tasks.
Effort: Medium
Relevant skills: coding-agent (installed, allowed on autonomous), github (installed, allowed on autonomous), gh-issues (installed, allowed on autonomous)
Note: This is the path toward "largely autonomous personal ops system." The guard rails are in place (PR-only, no main merges). Start with safe, idempotent tasks.

**[Policy override audit logging]**
Why: preflight.js supports `policyOverride=true` with a reason string, but there is no logging of when overrides are used. This is a governance gap -- Bryan should be able to see a trail of all policy overrides.
How: When `validateEnvelope` encounters a valid override, append to a memory/policy_overrides.log.md file with timestamp, hat, and reason. Include override count in the weekly ops report.
Effort: Low
Relevant skills: (no new skills -- change to os/preflight.js)
Note: This touches a dangerous file (os/preflight.js). Flag in PR description per CLAUDE.md rules.

**[Hat-scoped model chain optimization]**
Why: All five hats currently use nearly identical model chains (minor ordering differences). The system has 6 allowed models but no evidence of measuring which model performs best for which task type. The free-tier models (nemotron-nano, arcee-trinity-mini) are on the allowed list but may not be well-suited for all hats.
How: Add lightweight quality scoring to task outputs (did the task succeed? was output useful?) and use the data to tune per-hat model chains over time. Start by logging model selection per task in memory/.
Effort: High
Relevant skills: model-usage (installed, allowed on ops-core)
Note: Changes to os/model-routing.js are dangerous per CLAUDE.md.

**[Notion as a structured knowledge base]**
Why: The `notion` skill is installed and allowed on ops-core, but there is no integration. Notion could serve as a structured layer for resale inventory tracking, project status dashboards, or a curated link library -- things that don't fit well in Markdown files.
How: Create a Notion database for resale inventory (item, photos, comps, listing status, sold price). Have the resale pipeline write to it. Bryan gets a visual dashboard for free.
Effort: Medium
Relevant skills: notion (installed, allowed on ops-core)
Note: Would require adding notion to the resale hat's allowedSkills in os/hat-profiles.json -- this is a permission expansion and will trigger escalate-to-Bryan review.
