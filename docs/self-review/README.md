# docs/self-review/

Weekly self-review artifacts generated by Bort's `scripts/self-review-job.mjs` cron.

## Folder Structure

```
docs/self-review/
├── README.md          — this file
├── findings/          — per-run code review findings logs
├── research/          — per-run web research digests
└── plans/             — per-run forward project plans
```

## What Each Folder Contains

### findings/

One file per run: `YYYY-MM-DD-findings.md`

Lists all code review findings from that week's scan — every file group reviewed,
every finding (HIGH/MEDIUM/LOW), and the disposition (PR opened, logged only, skipped).

### research/

One file per run: `YYYY-MM-DD-digest.md`

Web research digest covering four topics: AI/agent tooling, eBay/resale market,
OpenClaw ecosystem, and general tech/stack news. Actionable items are flagged
with `[ACTION]`.

### plans/

One file per run: `YYYY-MM-DD-plan.md`

Forward-looking plan summarizing the run, open items, proposed next actions,
and questions for Bryan. Each plan references the previous plan file to maintain
continuity.

## Schedule

- **Full self-review:** Weekly, Sundays at 7:00 AM Phoenix time
- **PR staleness check:** Daily at 11:00 AM Phoenix time

## How It Works

`scripts/self-review-job.mjs` runs five phases:

1. **Code Review** — scans scripts/, integrations/, hats/, os/, skills/ for issues; also loads `.learnings/` (populated by `self-improving-agent`) as additional review context
2. **PR Staleness** — flags PRs open > 3 days (excluding WIP)
3. **Web Research** — researches four topic areas via Gemini
4. **Write Notes** — commits findings, digest, and plan to this directory
5. **Telegram Summary** — sends Bryan a completion report

Fix PRs (HIGH/MEDIUM findings) go through normal PR review. Notes are committed
on bort/ branches directly to main.

## Runtime Learnings Integration

The `self-improving-agent` ClawHub skill passively captures runtime failures, corrections,
and knowledge gaps to `.learnings/` on the VPS. During Phase 1, the self-review job reads
these learnings and injects them as additional context for the code reviewer, creating a
feedback loop:

```text
runtime failures/corrections → .learnings/ → weekly self-review → fix PRs
```

The `.learnings/` directory is gitignored (local to VPS only). See `.learnings/README.md`
for the file format and trigger conditions.
